The fix is a one-liner rewrite in the proxy config. Here's what was wrong and why:

  Root cause: The app prefixes all fetch paths with /api (e.g. /api/search/semantic) so Vite's proxy can intercept     them as a same-origin request. But the proxy was forwarding the full path including /api to the backend — hitting
  neuronx.jagatab.uk/api/search/semantic which doesn't exist. The backend routes are at /search/semantic,              /search/domains, etc. (no /api prefix on the server).

  Fix in nginx.conf: Changed proxy_pass https://neuronx.jagatab.uk/api/ → proxy_pass https://neuronx.jagatab.uk/ so

What the tool currently is
                                                                                                                       This frontend talks to one fixed backend — the NeuronX API at neuronx.jagatab.uk. That backend does the vector
  search. The UI (this tool) is just the interface layer. To make it work for any company's data, you swap or extend
  the backend.

  ---
  The two realistic paths

  Path A — Point it at a custom backend (fastest)

  A company runs their own search backend that exposes the same API shape this tool already calls:

  POST /api/search/v2/search   → returns { results, total, query_time_ms }
  POST /v1/chat/completions    → OpenAI-compatible SSE streaming

  Their stack would be:

  Company data → ingestion pipeline → vector DB → search API → this frontend

  Concretely:
  - Ingest: chunk documents into ~300-token pieces, embed them (OpenAI, Cohere, local model)
  - Store: Qdrant, Weaviate, Pinecone, or Chroma
  - Serve: a small FastAPI/Express layer that accepts the same request shape this frontend sends
  - LLM: any OpenAI-compatible endpoint (OpenAI, Azure OpenAI, Ollama locally, vLLM)

  The frontend needs zero code changes except two lines in vite.config.ts and nginx.conf pointing to the new backend
  URL.

  Path B — Multi-tenant SaaS (more work, bigger opportunity)

  One deployment serves many companies, each with isolated data:

  - Each company gets an API key and a namespace/collection in the vector DB
